## 🟢 What went well

**Murat** — Patterns qui ont payé :

- **Crate `kesh-qrbill` publishable** : refonte SIX 2.2 propre (validation IBAN/QR-IBAN/QRR mod-10, ISO-3166 alpha-2, cross-check QR-IBAN ↔ Reference type 4 cas exhaustifs SIX §3.3, Annex C charset strict + ajouts pragmatiques U+2019/U+2014/U+20AC). Réutilisable hors Kesh.
- **Golden test Plan C** : payload round-trip validé (pas de hash SHA-256 binaire — `printpdf` injecte un `/ID` aléatoire, aurait flappé). Décision pragmatique payante.
- **`validate_invoice` atomique** : invoice + sequence_lock + journal_entry créés dans **une seule transaction**, ordre de locks documenté (`invoices → fiscal_years → invoice_number_sequences → journal_entries`). Test concurrence `test_concurrent_validate_vs_manual_journal_entry` verrouille l'invariant.
- **Review 4 groupes × 11 passes (Story 5-4)** : pattern DB / API / Crate / Frontend parallélisé, Opus/Sonnet/Haiku orthogonaux. Sonnet/Haiku ont attrapé des régressions Opus invisibles à Opus lui-même (cf. F2 Auditor Group A « date_from/date_to perdus dans summary »).
- **Defense-in-depth** : CHECK DB `chk_invoices_validated_has_je` + validation `kesh-core::create_in_tx` + assertion test. Trois couches indépendantes.
- **i18n complet** : ~40 clés × 4 locales sur 5-4, zéro clé manquante en finale. Helper `formatSwissAmount` (Story 3-2) réutilisé dans tous les composants facture.

## 🟡 What could improve

**Alice** — À retenir pour la suite :

- **Dérogations spec actées en review** : AC#8 amendé (`paid_at` peut être futur — clarification domaine bancaire), CHECK DB élargie à `cancelled` (anticipation Epic 10→11 avoirs), sémantique `payment_status` precedence (N3 non spécifié). **Risque** : les specs doivent prévoir les dérogations probables (`paid_at` futur était une omission, pas une erreur).
- **Fixtures E2E non exécutées en CI** : `invoice_pdf_e2e` cassé depuis 5-3 (KF-001, 10/11 tests), détecté à chaud en code review 5-4, fixé manuellement le 2026-04-16. Sans CI gate, chaque story peut laisser passer une régression E2E silencieuse.
- **Contradictions spec entre stub et colonne réelle** : stub 5-4 mentionnait `value_date`, colonne réelle `due_date`. Corrigé dans la spec 1.0 mais a dû être rattrapé — piège récurrent des stubs backlog écrits tôt.
- **Playwright `invoices_echeancier.spec.ts` livré mais non exécuté** (requiert stack complète + seed). Zéro garantie d'exécution réelle — placé en dette.
- **Bug latents cross-story corrigés au passage** : `delete()` SELECT FOR UPDATE omettait `journal_entry_id`, tests `test_{update,delete}_rejects_non_draft` cassés par la CHECK `chk_invoices_validated_has_je` de 5-2. Trois bugs pré-existants découverts *en cours* de 5-4 → le filet de tests avant code-review n'est pas assez rigide.
- **Dettes D1 (multi-tenant `get_company`) et D2 (whitelist TVA DB-driven) non adressées** — transmises d'Epic 4 à Epic 5, vont encore se propager en Epic 7 (ex-6) Import bancaire. Cleanup multi-tenant devient urgent avant l'import qui va multiplier les scopes `company_id`.

## 🔴 What was hard / pénible

**Guy** :

- **Beaucoup de review pour atteindre le critère « rien au-dessus de LOW »**. Story 5-4 a demandé 11 passes adversariales réparties sur 4 groupes pour converger. Coût cognitif et LLM élevé. Signal que la story était trop large — 5-4 couvrait échéancier + export CSV + mark-paid en un seul bloc. **Règle retenue** : splitter plus agressivement dès la phase spec. Si une story déclenche plus de 4 passes de review pour converger, c'est un signal rétrospectif qu'elle aurait dû être splittée.
